# Requirements Document

## Introduction

The Aegis Suite currently ships as a self-hosted application: each end user is asked to register their own Discord application, paste their own bot token into a Setup Wizard, and run the FastAPI dashboard locally. The intended deployment model is the opposite — the maintainer hosts a single bot and a single dashboard instance, end users invite that bot to their server, run `/linkdashboard` to receive a 6-digit pairing code, and paste the code into the hosted dashboard to unlock their tenant view.

This feature migrates the codebase from the self-hosted model to the managed-hosting model. The work covers six concrete areas:

1. Strip every user-facing affordance for configuring a bot token (UI, frontend logic, REST endpoints, offline screens, README).
2. Repair `build_exe.py` so it produces a single executable plus an installable desktop shortcut (OneDrive-aware) for the single user who still wants a local desktop build (the maintainer).
3. Make `run.py` safe to import on headless cloud hosts (Railway, Render) where `webbrowser.open` would crash.
4. Provide a `requirements.txt` at the repository root so cloud installers (`pip install -r requirements.txt`) succeed.
5. Add a GitHub Actions workflow that runs the test suite and deploys the project to Railway.
6. Rewrite the Discord Bot Setup section of `README.md` to describe the invite + `/linkdashboard` flow.

The spec must not modify any of the existing security, isolation, rate-limiting, or session-revocation behaviors that already protect tenants.

## Glossary

- **Aegis_Suite**: The combined Python Discord bot (`bot_manager.py`) and FastAPI dashboard (`web_server.py`) under deployment.
- **Maintainer**: The developer who hosts the single production instance of Aegis_Suite and operates the `.env` file on the server.
- **Tenant**: An end user (Discord server administrator) who invites the hosted bot to their guild and unlocks the dashboard via a 6-digit pairing code.
- **Pairing_Code**: The 6-character ephemeral, alphanumeric, single-use code generated by the `/linkdashboard` slash command and stored under `pending_pairings` in `config.json` with an `expires_at` timestamp.
- **Bot_Token**: The Discord bot credential (`DISCORD_BOT_TOKEN`) read from the server-side `.env` file by `utils.get_bot_token`. It is a maintainer secret and must never reach the browser.
- **Dashboard_UI**: The single-page application served from `static/index.html` and `static/app.js`.
- **Setup_Wizard**: The DOM element `#setup-wizard` in `static/index.html` that prompts the user for a Bot_Token and Client ID.
- **Reconfigure_Button**: The DOM element `#btn-reconfigure` in `static/index.html` that lets a logged-in user wipe `bot_token` and re-enter the Setup_Wizard.
- **Bot_Toggle_Button**: The DOM element `#btn-bot-toggle` in `static/index.html` that triggers `/api/bot/start` and `/api/bot/stop`.
- **Offline_Notice**: The DOM element `#offline-notice-overlay` in `static/index.html` shown when a non-admin visitor lands on the dashboard while the bot process is not running.
- **Build_Script**: `build_exe.py`, the PyInstaller wrapper that produces a Windows distribution.
- **Launcher_Script**: `run.py`, the bootstrap script that creates `.venv`, installs dependencies, and starts uvicorn.
- **Cloud_Host**: A headless Linux container environment such as Railway or Render where no display server or browser is available.
- **Deploy_Workflow**: The GitHub Actions workflow file at `.github/workflows/deploy.yml`.
- **Verify_Workflow**: The pre-existing GitHub Actions workflow at `.github/workflows/verify.yml`.
- **Tenant_Session**: A signed JWT session created by `auth.create_session(guild_id, "tenant")` containing `{guild_id, role, exp}`, valid for 24 hours.
- **Pairing_Onboarding_Flow**: The end-user flow: invite the hosted bot, run `/linkdashboard`, copy the Pairing_Code, paste it into the dashboard login field.

## Requirements

### Requirement 1: Remove the Setup Wizard and credential entry UI

**User Story:** As a Tenant, I want to never see a bot-token form, so that I am not asked for credentials I do not own.

#### Acceptance Criteria

1. THE Dashboard_UI SHALL omit the `#setup-wizard` element entirely from `static/index.html`.
2. THE Dashboard_UI SHALL omit the `#wizard-token` input, the `#wizard-client-id` input, and the `#btn-save-wizard` button from `static/index.html`.
3. THE Dashboard_UI SHALL omit the `#btn-reconfigure` button and its surrounding "Setup Configuration" / "Change Credentials" info row from the Bot Connection Info card in `static/index.html`.
4. THE Dashboard_UI SHALL omit the `#btn-bot-toggle` element from the sidebar bot badge in `static/index.html`.
5. WHEN `static/app.js` loads in the browser, THE Dashboard_UI SHALL NOT register a click listener for `#btn-save-wizard`, `#btn-bot-toggle`, or `#btn-reconfigure`.
6. WHEN `static/app.js` loads in the browser, THE Dashboard_UI SHALL NOT define or call `saveWizardCredentials`.
7. WHEN the Dashboard_UI renders the login overlay, THE Dashboard_UI SHALL display the Pairing_Onboarding_Flow steps (invite the bot, run `/linkdashboard`, copy the 6-digit code, paste it, click Unlock Dashboard).

### Requirement 2: Remove server-side endpoints that accept or expose the bot token

**User Story:** As the Maintainer, I want the bot token to live only in the server's `.env`, so that no browser request can read or rewrite it.

#### Acceptance Criteria

1. THE Aegis_Suite SHALL remove the `POST /api/bot/start` endpoint from `web_server.py`.
2. THE Aegis_Suite SHALL remove the `POST /api/bot/stop` endpoint from `web_server.py`.
3. WHEN a request is sent to `/api/bot/start` or `/api/bot/stop`, THE Aegis_Suite SHALL respond with HTTP 404.
4. THE Aegis_Suite SHALL remove the `bot_token` field from the `ConfigModel` Pydantic model in `web_server.py`.
5. WHEN `GET /api/config` is invoked, THE Aegis_Suite SHALL NOT include any `bot_token` key in the response body.
6. WHEN `POST /api/config` is invoked with a body containing a `bot_token` field, THE Aegis_Suite SHALL ignore that field and SHALL NOT write its value to `.env`, `os.environ`, or `config.json`.
7. THE Aegis_Suite SHALL load the Bot_Token exclusively via `utils.get_bot_token`, which reads `DISCORD_BOT_TOKEN` from `os.environ` populated by the server-side `.env` file.
8. WHEN the FastAPI lifespan startup runs and `utils.get_bot_token` returns a non-empty value, THE Aegis_Suite SHALL invoke `bot_manager.start_bot_service(token)`.
9. IF the FastAPI lifespan startup runs and `utils.get_bot_token` returns an empty value, THEN THE Aegis_Suite SHALL log a maintainer-facing error and SHALL NOT prompt any browser client to supply a token.
10. THE Aegis_Suite SHALL retain the existing `%%BOT_API_URL%%` sentinel injection performed by the root HTML handler in `web_server.py`.

### Requirement 3: Replace the offline experience with a maintenance message

**User Story:** As a Tenant, I want a clear maintenance message when the hosted bot is down, so that I am not confused by setup instructions that do not apply.

#### Acceptance Criteria

1. WHEN `GET /api/status` returns `status` equal to `"stopped"` or `"connecting"`, THE Dashboard_UI SHALL display the Offline_Notice overlay.
2. THE Offline_Notice SHALL display a single human-readable maintenance message stating that the dashboard is temporarily unavailable and that the user should try again later.
3. THE Offline_Notice SHALL NOT contain any link, button, or instruction that asks the user to provide a Bot_Token, Client ID, or any other credential.
4. THE Offline_Notice SHALL NOT contain any control that triggers `/api/bot/start` or `/api/bot/stop`.

### Requirement 4: Eliminate browser-side persistence of credential fields

**User Story:** As a Tenant, I want my browser to never cache a bot token, so that residual state from earlier versions cannot reappear.

#### Acceptance Criteria

1. THE Dashboard_UI SHALL NOT read or write any `localStorage` or `sessionStorage` key whose name contains `token`, `bot_token`, or `client_id`, with the explicit exception of the existing dashboard session JWT (`auth_token`).
2. WHEN the Dashboard_UI loads for the first time after this migration, THE Dashboard_UI SHALL delete any pre-existing `localStorage` keys named `bot_token` or `client_id` if present, to clear residual state from prior versions.
3. THE Dashboard_UI SHALL NOT include any DOM element whose `id` or `name` attribute equals `wizard-token`, `wizard-client-id`, or `bot-token`.

### Requirement 5: Produce a single-file Windows executable

**User Story:** As the Maintainer, I want `build_exe.py` to emit one `AegisOptimizer.exe`, so that local distribution is a single file rather than a directory tree.

#### Acceptance Criteria

1. WHEN `python build_exe.py` is executed on Windows, THE Build_Script SHALL invoke PyInstaller with the `--onefile` flag.
2. WHEN `python build_exe.py` completes successfully, THE Build_Script SHALL produce the file `dist/AegisOptimizer.exe`.
3. WHEN `python build_exe.py` is executed on Windows, THE Build_Script SHALL pass `--icon logo.ico` to PyInstaller, using the icon converted from `bot_logo.png` by the existing `convert_png_to_ico` step.
4. WHEN `python build_exe.py` is executed on Windows, THE Build_Script SHALL bundle the `static` and `templates` directories via `--add-data`.
5. IF PyInstaller exits with a non-zero return code, THEN THE Build_Script SHALL print the captured stdout and stderr and SHALL exit with a non-zero status code.
6. THE Build_Script SHALL NOT continue with shortcut creation when PyInstaller has failed.

### Requirement 6: Create a desktop shortcut for the compiled executable

**User Story:** As the Maintainer, I want a desktop shortcut created automatically after a successful build, so that I do not need a separate Inno Setup installer.

#### Acceptance Criteria

1. WHEN `python build_exe.py` completes a successful PyInstaller `--onefile` build on Windows, THE Build_Script SHALL create a Windows shortcut named `Aegis Optimizer.lnk` on the current user's Desktop.
2. THE shortcut SHALL set its target to the absolute path of `dist/AegisOptimizer.exe`.
3. THE shortcut SHALL set its working directory to the directory containing `dist/AegisOptimizer.exe`.
4. THE shortcut SHALL set its icon to the absolute path of `logo.ico`.
5. WHEN determining the Desktop folder path, THE Build_Script SHALL use `winshell.desktop()` and SHALL accept a OneDrive-redirected path (e.g. `C:\Users\<name>\OneDrive\Desktop`) as a valid Desktop folder.
6. WHEN `winshell` or `pywin32` (`win32com.client`) is missing, THE Build_Script SHALL print a warning naming both packages and SHALL exit with status 0 without creating the shortcut, leaving the compiled executable in place.
7. IF the resolved Desktop folder does not exist on disk, THEN THE Build_Script SHALL log an error including the resolved path and SHALL skip shortcut creation without failing the overall build.
8. THE Build_Script SHALL NOT depend on Inno Setup or any external installer toolchain to create the desktop shortcut.

### Requirement 7: Make `run.py` safe to start on headless cloud hosts

**User Story:** As the Maintainer, I want `run.py` to skip browser-launching on Railway and Render, so that the process does not crash on a headless container.

#### Acceptance Criteria

1. WHEN `run.py` is executed and neither the environment variable `RAILWAY_ENVIRONMENT` nor `RENDER` is set to a non-empty value, THE Launcher_Script SHALL call `webbrowser.open("http://127.0.0.1:8000")` after a 1.5-second delay.
2. WHEN `run.py` is executed and either `RAILWAY_ENVIRONMENT` or `RENDER` is set to a non-empty value, THE Launcher_Script SHALL NOT call `webbrowser.open` and SHALL NOT spawn the background browser-opening thread.
3. IF `webbrowser.open` raises any exception, THEN THE Launcher_Script SHALL log a warning and SHALL continue running the uvicorn server.
4. THE Launcher_Script SHALL continue to launch uvicorn on host `127.0.0.1` port `8000` regardless of whether the browser was opened.

### Requirement 8: Provide `requirements.txt` at the repository root

**User Story:** As the Maintainer, I want a `requirements.txt` checked in at the repo root, so that `pip install -r requirements.txt` works on Railway, Render, and any standard PaaS.

#### Acceptance Criteria

1. THE Aegis_Suite SHALL include a `requirements.txt` file at the repository root.
2. THE `requirements.txt` SHALL list, at minimum, the runtime dependencies currently installed by `run.py`: `discord.py`, `fastapi`, `uvicorn`, `websockets`, `yt-dlp`, `PyNaCl`, and `pydantic`.
3. THE `requirements.txt` SHALL pin each dependency to an explicit version specifier (`==` or `~=`) rather than leaving the version unbounded.
4. WHEN a fresh virtual environment runs `pip install -r requirements.txt` on Python 3.12, THE installation SHALL complete with exit code 0.

### Requirement 9: GitHub Actions deploy workflow with a test job and a deploy job

**User Story:** As the Maintainer, I want pushes to `main` to run the test suite and then deploy to Railway, so that I do not deploy unverified code.

#### Acceptance Criteria

1. THE Aegis_Suite SHALL include a workflow file at `.github/workflows/deploy.yml`.
2. THE Deploy_Workflow SHALL trigger on `push` events to the `main` branch.
3. THE Deploy_Workflow SHALL define a job named `test` that runs `pytest` against the `tests/` directory on `ubuntu-latest` using Python 3.12 after installing dependencies from `requirements.txt`.
4. THE Deploy_Workflow SHALL define a job named `deploy` that depends on the `test` job (`needs: test`) and runs only when the `test` job has succeeded.
5. THE `deploy` job SHALL install the Railway CLI and SHALL invoke `railway up` (or the equivalent non-interactive deploy command) using a `RAILWAY_TOKEN` secret read from `secrets.RAILWAY_TOKEN`.
6. IF `secrets.RAILWAY_TOKEN` is unset at workflow runtime, THEN THE Deploy_Workflow SHALL fail the `deploy` job with a clear error message and SHALL NOT attempt to call `railway up`.
7. THE Deploy_Workflow SHALL NOT replace, delete, or disable the existing `.github/workflows/verify.yml` Verify_Workflow.

### Requirement 10: Rewrite the Discord Bot Setup section of the README

**User Story:** As a Tenant reading the README, I want a four-step invite-and-link guide, so that I never look for a Discord Developer Portal step that no longer applies.

#### Acceptance Criteria

1. THE `README.md` SHALL replace the existing "🤖 Discord Bot Setup Instructions" section with a section describing the Pairing_Onboarding_Flow.
2. THE rewritten section SHALL list, in order: (a) click the Invite button to add Aegis to the user's Discord server, (b) run `/linkdashboard` in the user's Discord server, (c) paste the resulting 6-digit code into the hosted dashboard, (d) confirmation that the server panel is now unlocked.
3. THE `README.md` SHALL NOT instruct end users to visit the Discord Developer Portal, create an Application, enable Privileged Gateway Intents, copy a Bot Token, or paste credentials into a Setup Wizard.
4. THE `README.md` SHALL include a placeholder (for example `[your domain]`) where the Maintainer's hosted dashboard URL should appear.
5. THE `README.md` SHALL retain the existing "⚠️ Known Technical Debt & Limits" section unchanged, including its references to JSON file contention and the SQLite migration roadmap.

### Requirement 11: Preserve all existing security, isolation, and session-revocation behaviors

**User Story:** As a Tenant, I want every existing protection (rate limiting, guild isolation, signed sessions, ReDoS guards, XSS escaping) to remain in force after the migration, so that this refactor does not weaken security.

#### Acceptance Criteria

1. THE Aegis_Suite SHALL retain the `/linkdashboard` slash command in `bot_manager.py`, including the writes to `pending_pairings` in `config.json` with `expires_at` timestamps and the `can_generate_code` rate guard.
2. THE Aegis_Suite SHALL retain `auth.create_session`, `auth.decode_token`, `auth.validate_session`, and the `{guild_id, role, exp}` JWT payload contract in `auth.py`.
3. THE Aegis_Suite SHALL retain the `hmac.compare_digest` constant-time signature comparison in `auth.decode_token`.
4. THE Aegis_Suite SHALL retain the guild-scoped `auth_middleware` checks in `web_server.py` that match `/api/guilds/(\d+)` against `auth.get_session_guild_id` for tenant sessions.
5. THE Aegis_Suite SHALL retain the per-guild sliding-window rate limiter (60 requests per minute) and the background `prune_stale_rate_limiters` task started in the FastAPI lifespan.
6. THE Aegis_Suite SHALL retain the `/unlink` and `/unlink purge` slash commands and the `on_guild_remove` event handler that invokes `auth.revoke_guild_sessions`.
7. THE Aegis_Suite SHALL retain `utils.is_regex_safe` and its usage in `bot_manager.py` auto-responder evaluation and in the `/api/auto-responders` POST and PUT endpoints.
8. THE Aegis_Suite SHALL retain the `escapeHtml` helper in `static/app.js` and all of its current call sites.
9. THE Aegis_Suite SHALL retain the `utils.config_lock` and `utils.giveaways_lock` `asyncio.Lock` write guards on `config.json` and `giveaways.json` writes.
10. THE Aegis_Suite SHALL retain the `.gitignore` exclusions for `.env`, `config.json`, `giveaways.json`, `audit_log.json`, and `leveling_data.json`.
11. THE Aegis_Suite SHALL retain the `admin` and `platform_owner` Maintainer roles and the `tenant` role with their current authorization scopes.
12. THE Aegis_Suite SHALL leave the existing TODO comments concerning the SQLite migration, schema versioning, and unified `check_auth_permission` authorization matrix in place without addressing them in this spec.
